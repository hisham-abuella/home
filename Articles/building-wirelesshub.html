Building WirelessHub with Claude

February 16, 2026 6 min read Hisham Abuella

How I helped build a wireless engineering blog from scratch using Claude Code as my AI pair programmer

A few months ago, my friend Ahmed Elshafie — a Wireless Systems Engineer at Apple and IEEE Senior Member — wanted a place to share his deep knowledge of wireless communications, signal processing, and mathematics. He had the expertise and the articles. What he needed was a home for them on the web. That's where I came in, and that's where Claude Code became my co-builder.

The Vision

The goal was straightforward: build a clean, fast, content-focused blog called WirelessHub that would make Ahmed's technical writing shine. No heavy frameworks. No build tools. No npm. Just pure HTML, CSS, and vanilla JavaScript — a static site that loads instantly and works everywhere.

Ahmed already had 25+ articles covering everything from FFT in everyday life to MIMO beamforming to quantum wireless communication. The site needed to organize all of that with search, filtering, categorization, dark mode, and even embedded podcast episodes.

Working with Claude Code

Claude Code was not just an autocomplete tool on this project. It was a genuine pair programmer. Here's how the workflow looked in practice:

Architecture Claude helped design the tab-based layout, the JSONL search database, and the sidebar widget system
Implementation From the dark mode toggle to the podcast player with scrubbing and progress bars — Claude wrote production-ready code
Search System A live keyword search powered by a JSONL database fetched at runtime, matching across titles, tags, and categories
Design Polish Hover animations, card shadows, responsive layouts, and a cohesive color system across light and dark themes

I would describe what I wanted — sometimes precisely, sometimes loosely — and Claude would explore the codebase, understand the existing patterns, and make changes that fit naturally. When I said "add GoatCounter analytics to all pages," Claude updated 27 HTML files, added the tracking script, created an article template, and updated the project documentation, all in one pass.

The best part of working with Claude Code is that it reads before it writes. It understands the codebase first, then makes changes that respect existing conventions.

What We Built

The final site is entirely static — hosted for free on GitHub Pages — yet it feels like a full-featured content platform:

Tab navigation with Home, Articles, Podcasts, and About sections. Category filtering with chips and sort buttons. A featured article hero on the home page. Podcast players in both the sidebar and a dedicated tab, with play/pause, progress bars, and scrubbing. Dark mode that persists across sessions. A live search that filters articles and podcasts by keyword. And privacy-friendly analytics via GoatCounter.

All of it in vanilla HTML, CSS, and JS. No build step. No dependencies. Just clean, readable code.

Lessons Learned

AI pair programming is real

Claude Code didn't just generate code snippets — it understood context. It read existing files, matched coding styles, and made multi-file changes that were consistent. It caught things I missed and suggested approaches I hadn't considered.

Static sites still win

No framework fatigue. No dependency updates. No build failures. The site loads fast, works offline after first load, and will keep working for years without maintenance. For a content-focused blog, this is exactly right.

Documentation is a force multiplier

We maintained a detailed CLAUDE.md file that documented every convention — the tab system, the JSONL format, the card HTML structure, the category CSS classes. This meant that every time I started a new session with Claude, it could immediately understand the project and make correct changes without re-explaining anything.

Good documentation doesn't just help humans. It makes AI collaboration dramatically more effective.
